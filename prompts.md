## Prompt 8
Initialize the Django 5 + DRF + PostgreSQL backend (`/backend`) from scratch: project `foresight_backend`, `django-environ` config, a `core` app with a DRY `BaseModel` (UUID pk + soft delete), multi-tenant `Organization` and `CustomUser` models, and a matching `apiClient.ts` foundation on the frontend.

### Architectural Decisions
- **Settings via `django-environ`**: `foresight_backend/settings.py` reads `SECRET_KEY`, `DEBUG`, `ALLOWED_HOSTS`, `DB_*`, and `CORS_ALLOWED_ORIGINS` from `.env` (gitignored; `.env.example` committed). No credentials are hardcoded anywhere in source.
- **Local Postgres via Docker Compose**: `backend/docker-compose.yml` runs `postgres:16-alpine` with credentials matching `.env`'s defaults, so `docker compose up -d && python manage.py migrate` works out of the box for any engineer cloning the repo. Chosen over a bare-metal local install per user preference (2026-07-18).
- **Soft delete, DRY**: `core.models.BaseModel` is an abstract base (UUID `id`, `created_at`, `updated_at`, `deleted_at`) shared by every tenant-scoped model. `.delete()` stamps `deleted_at` instead of issuing SQL `DELETE`; `.hard_delete()` exists for the rare permanent-removal case. `objects` (the default manager, by declaration order) excludes soft-deleted rows; `all_objects` includes them.
- **Multi-tenancy**: `Organization` is the tenant root. `CustomUser(AbstractUser, BaseModel)` carries `organization` (nullable only for platform superusers) and `is_org_admin`. `AUTH_USER_MODEL = "core.CustomUser"`. A parallel `CustomUserManager` preserves Django's `create_user`/`create_superuser` while adding the same soft-delete filtering.
- **Ruff DJ012 suppressed intentionally**: the linter's manager-ordering heuristic misreads `all_objects` (declared after `objects`) as a misplaced field. Declaration order there is load-bearing -- Django uses the first-declared manager as the implicit default -- so the rule is disabled repo-wide in `pyproject.toml` rather than reordered.
- **Phase 1 auth posture**: `REST_FRAMEWORK.DEFAULT_PERMISSION_CLASSES` is `AllowAny` for now, per CLAUDE.md's phased security note -- models and the user/org relationship are already RBAC-ready (`is_org_admin`, `is_superuser`, tenant FK) so tightening to `IsAuthenticated` + tenant-scoped querysets later is a permissions-layer change, not a schema change.
- **`frontend/lib/apiClient.ts`**: single fetch-based `ApiClient` class (base URL from `NEXT_PUBLIC_API_URL`), JSON-in/JSON-out, optional `Idempotency-Key` header support, and an `ApiError` that carries the parsed DRF field-error payload (`Record<string, string[]>`) for inline form error mapping. Intentionally separate from the existing `services/api.ts` mock data layer, which still backs the current UI until real endpoints exist.

## Prompt 9
Build the `rules` Django app end-to-end (model, strict serializer validation, idempotent DRF view) and wire the existing Next.js Rule Builder form and a new rules list screen to the real backend, replacing the mock multi-condition form.

### Architectural Decisions
- **`HealthRule(BaseModel)`**: FK to `Organization` (`related_name="health_rules"`), `name`, `metric_type` (a `TextChoices` mirroring `customers.EventLog.EventType`'s vocabulary so the future scoring engine can filter the same telemetry by name), `threshold` (`DecimalField`, `>= 0`), `weight` (`PositiveSmallIntegerField`, `1-100`), `is_active`. `name` and `is_active` weren't in the original field list but were added because a rules *list* screen is unusable without an identifiable, toggleable row -- both are DRY with the same `unique_active_<x>_per_org` soft-delete-aware constraint pattern `Customer` already uses.

## Prompt 10
Build the `HealthScoreEngine` (the Churn Scoring Engine from CLAUDE.md): a `services.py` that evaluates an Organization's `HealthRule`s against `Customer`/`EventLog` telemetry, an API endpoint to trigger it, and a "Recalculate Health Score" button in the Next.js Customer 360 view.

### Architectural Decisions
- **`customers/services.py`, not `rules/services.py`**: the engine's output (`Customer.health_score`) and primary query root (`Customer`) both live in `customers`; it only *reads* `HealthRule` from `rules`. Keeping it next to the model it mutates avoided a cross-app import cycle in the other direction (the alternative would need `rules` to import `Customer`).
- **Violation direction per metric type is a business-logic call the model/serializer never encoded** -- `HealthRule.threshold` is just a number with no stored sign/direction. Rather than adding a schema field (out of scope, and the seeded demo rules already imply direction by name), `services.py` hardcodes two sets: `LOW_VOLUME_METRICS` (`login`, `feature_usage` -- violated when recent count `< threshold`, i.e. too little engagement) and `HIGH_VOLUME_METRICS` (`support_ticket`, `billing_issue`, `contract_signal` -- violated when count `>= threshold`, i.e. friction piling up). `nps_response` is handled separately: it compares `Customer.nps` directly against the threshold rather than counting `EventLog` rows, since NPS is a stored score, not an event tally. This mirrors the demo rules in `seed_demo_data.py` (e.g. "Login drop watchlist" @ threshold 2, "Support ticket spike" @ threshold 3).
- **30-day rolling evaluation window** (`EVALUATION_WINDOW_DAYS`): EventLog volume is judged against recent behavior, not all-time history, so a customer who logged in heavily a year ago doesn't get a free pass today. No such window existed in any prior spec, so this was chosen as a reasonable, documented default rather than left unbounded.
- **No N+1, one query per run**: `HealthScoreEngine._load_customers` annotates one `Count(..., filter=Q(...))` per *distinct metric type an active rule actually references* directly on the `Customer` queryset (a single `LEFT JOIN` + `GROUP BY`), instead of looping customers and querying `EventLog` per row. Score mutation is written back with a single `Customer.objects.bulk_update(..., ["health_score", "updated_at"])` call (bulk_update bypasses `auto_now`, so `updated_at` is set explicitly in the loop). Covered by `test_run_only_queries_customers_in_target_organization`, which asserts a flat 3-query ceiling (rules select, annotated customer select, bulk update) regardless of customer count.
- **`POST /api/v1/customers/{id}/calculate/`**: a `CustomerViewSet` detail action (not a new top-level route) that calls `HealthScoreEngine(customer.organization).run(customer_ids=[customer.id])` and returns the full `CustomerDetailSerializer` payload at `200 OK` (not `201`/`204` -- it's a compute-and-return action, not create/delete, per CLAUDE.md ##3's RESTful status code rule). The engine itself already supports whole-org runs (`run()` with no `customer_ids`) for future batch/cron use without a second endpoint.
- **Frontend wiring reuses the existing remediation-button pattern** in `Customer360` (`components/features/customer-360.tsx`): a new `recalculateCustomerHealth` fetcher (`services/api.ts`) and `recalculateHealthScoreAction` Server Action (`app/actions.ts`) that revalidates `/dashboard`, `/dashboard/accounts`, and the customer detail path -- mirroring `updateCustomerHealthAction`'s shape so the component has one consistent async-action-plus-toast pattern rather than two. The health gauge, tier badge, and "Product adoption" signal bar now read from a `currentScore` state (previously the gauge read `customer.engagementScore` directly, which never updated after the initial server render).
- **Verified against a live Postgres-backed run**, not just pytest: seeded demo data, hit the real endpoint through the actual UI (headless Chrome via CDP, since `chromium-cli`/Playwright weren't available offline), and confirmed a customer's score moved 47 → 45 in both the UI and a direct `GET /api/v1/customers/{id}/` call after clicking the button.

### Architectural Decisions

That rule protects the audited lifecycle of real tenant data reached through the API; this command targets a scratch/staging database being wiped and regenerated from scratch by an operator, not user-facing deletion. Deletion order is FK-safe (`EventLog` → `Customer` → `HealthRule` → `CustomUser` → `Organization`) since all four child models `PROTECT` their `organization` FK. Gated behind a `Type 'yes' to continue` prompt, skippable with `--noinput` for CI/automation.

Same approach for the 10,000 EventLogs. Verified `bulk_create` correctly populates `auto_now_add`/`UUID` PKs on this Postgres backend before relying on it at scale.
- **EventLog volume distributed with organic variance (`events._distribute_counts`), not an identical count per customer**: starts from an even split of the 10,000-event budget, then does a bounded number of random small transfers between customers (sum-preserving) so some accounts end up disengaged and others heavily active. Combined with a 90-day lookback against the engine's 30-day evaluation window, this produced a real spread on the first run (e.g. primary org: 128 Healthy / 101 At-Risk / 21 Critical out of 250) instead of every customer landing on the same score.

### Multi-Tenant Customer Ingestion (backend + frontend)
- **`core/tenancy.py::resolve_organization` gained real `X-Organization-ID` header support**, not just documentation of intent. Previously anonymous requests (all of them today — Phase 1 has no JWT) always fell back to the single earliest-created active Organization with no way to address a different tenant. Now: authenticated non-superuser → own org (unchanged); authenticated superuser → `None`/bypass (unchanged); anonymous → `Organization.objects.filter(id=<header>, is_active=True).first()` if the header is present and a syntactically valid UUID, else the same earliest-active-org fallback as before. `resolve_write_organization` (used by `CustomerViewSet.perform_create`) inherits this for free since it just wraps `resolve_organization`.
